# アーキテクチャ コスト評価とトレードオフ

> 本ドキュメントの前提となるプロダクト要件は [`app_overview.md`](./app_overview.md) を参照。

## 結論

**現アーキテクチャ (EKS ベース、3 AZ 構成) の月額固定費 $460〜560 は、「立ち上げ期は明確に過剰、スケール期 (MAU 10 万級) に達すれば回収可能なライン」** という二層構造で評価する必要がある。

- **K8s 学習が第一目的** の現フェーズでは、コスト回収観点での評価よりも「学習投資」として割り切るのが健全
- 想定ユーザーモデル (**社会人 + 学生 = 約 7000 万人潜在**) を字面通り受け止めれば、**スケール期到達時にはフリーミアム成立で十分に回収可能** な規模感
- ただし立ち上げ期 (MAU 数千〜数万) には固定費負担が重いため、**「触らない時は destroy」 + Spot ノード等の節約策を併用** すべき
- **学習目的を外した場合の最適解は単一サービスではなく、`ECS Fargate (api) + Lambda (batch/job)` のハイブリッド構成** ($300/月程度)。EKS の約 55% に圧縮しつつ production-grade な構成を維持できる

本ドキュメントでは以下を整理する:

1. 想定ユーザーモデルとトラフィック前提
2. 現アーキテクチャの月額コスト概算
3. 収益化シナリオと回収可能性
4. オーバーエンジニアリング判定 (ライフサイクル別)
5. コストを抑える運用上の工夫 (EKS 維持時)
6. 学習目的を外した場合の代替案 (4 パターン比較)

## 2. 現アーキテクチャの月額コスト概算

`system_architecture_diagram.drawio.png` から読み取った構成をもとに、東京リージョン (ap-northeast-1) で **本番として最低限動く最小構成** を組んだ場合の概算。

### 2.1 構成概要 (アーキテクチャ図 + Terraform 変数定義より)

## 3. 収益化シナリオと回収可能性

### 3.1 想定収益化モデル

| モデル | 現実性 | 想定 ARPU | 課金率仮定 |
|--------|------|---------|---------|
| **フリーミアム (基本無料 + プレミアム機能)** | ◎ | ¥500/月 | 無料の 1〜3% |
| **広告収益** | △ | ¥0.5〜2/PV | - |
| **B2B (採用企業向け検索・分析)** | ○ | ¥10万〜50万/社/月 | 営業力次第 |

→ 個人プロダクト主体のため、**フリーミアム + 一部広告** が現実的。

### 3.2 損益分岐の試算

#### 立ち上げ期 (MAU 1万)

- 課金ユーザー (1〜3%): 100〜300 人
- 月間収益: ¥50,000〜150,000 ≒ $330〜1,000
- **EKS 固定費 $464〜564 に対して、ほぼ赤字運用**

#### 中期 (MAU 10万)

- 課金ユーザー: 1,000〜3,000 人
- 月間収益: ¥500,000〜1,500,000 ≒ $3,300〜10,000
- 広告収益も加算 → **EKS 固定費を十分回収可能 (粗利 70〜95%)**

#### スケール期 (MAU 35万)

- 課金ユーザー: 3,500〜10,500 人
- 月間収益: ¥1,750,000〜5,250,000 ≒ $11,600〜35,000
- インフラ増強で $800〜1500/月に拡大しても、**粗利 90% 維持可能**

### 3.3 結論

**MAU 10 万を超えれば、現アーキテクチャでも回収可能** な規模感に到達する。一方で **MAU 10 万到達までの立ち上げ期は確実に赤字運用** であり、その期間をどう資金的に支えるかが論点になる。

ただし、本プロジェクトは **K8s 学習を第一目的** とするため、**「立ち上げ期の赤字 = 学習投資」** と整理するのが本来の位置付けに近い。

## 4. オーバーエンジニアリング判定 (ライフサイクル別)

### 4.1 判定マトリクス

| フェーズ | MAU 規模 | プロダクト目的での判定 | 学習目的での判定 |
|---------|---------|------------------|---------------|
| **立ち上げ期** | 〜1万 | **過剰** | 正当 (K8s 学習価値が最大) |
| **中期** | 1〜10万 | やや過剰 | 正当 |
| **スケール期** | 10〜35万 | **適正** (EKS の機能を実利用) | 正当 |
| **大規模** | 35万+ | 適正 | - (学習段階を超える) |

### 4.2 EKS が「過剰でなくなる」条件

スケール期到達時、EKS は以下の機能で価値を発揮する:

- **HPA / KEDA によるキュー長ベースのスケール**: 春就活シーズン等のスパイクに自動追従
- **Cluster Autoscaler / Karpenter**: ノードを需要に応じて増減 (Spot 活用で大幅削減)
- **Multi-AZ + マルチ Pod 配置**: 可用性要件を満たす
- **マイクロサービス分割の余地**: 機能追加に応じて Deployment を増やせる

これらの機能を **実トラフィックで使う** 段階に達すれば、EKS 採用は技術的にも経済的にも正当化される。

### 4.3 ただし学習目的としては全フェーズで正当

本プロジェクトの第一目的は **K8s (EKS) に習熟すること** であり、トラフィック規模に依存せず正当化される:

- **EKS Control Plane**: K8s API・kubectl・マニフェスト運用の土台
- **複数 Deployment / CronJob**: 各種ワークロードリソースの体験
- **Ingress + AWS Load Balancer Controller**: IRSA・LBC・Service の連携
- **マイクロサービス的分割**: Pod 間通信・Service Discovery の学習機会

→ 「過剰さ自体が学習価値」という構造のため、**立ち上げ期のコスト = 学習投資** として割り切る整理が健全。

### 4.4 学習投資としてのコスト評価

| 比較対象 | 概算費用 |
|---------|--------|
| **本構成を 6 ヶ月運用 (節約策なし)** | 約 $2,400〜3,000 (35〜45 万円) |
| **本構成を 6 ヶ月運用 (節約策あり)** | 約 $600〜1,200 (9〜18 万円) |
| CKA / CKAD 試験受験料 (1 回) | 約 $395 |
| Udemy / A Cloud Guru の K8s コース | 数千円〜2 万円 |
| Pluralsight 年間契約 | 約 5 万円 |

「実運用環境を自分で構築・運用する経験」は座学の数倍以上の価値があるが、**節約策を併用しなければ個人として重い投資**になる。

## 5. コストを抑える運用上の工夫 (EKS 維持時)

学習目的を維持しつつ、立ち上げ期の月額を抑える方法。

### 5.1 すぐ効くやつ

| 施策 | 削減見込み | トレードオフ |
|-----|----------|-----------|
| **NAT Gateway を 1 AZ に集約 (3 → 1)** | $66/月 | 残り 2 AZ の Pod が NAT 障害時に外部疎通不可 |
| **NAT Gateway を 2 AZ に減らす (3 → 2)** | $33/月 | 残り 1 AZ の Pod が NAT 障害時に外部疎通不可 |
| **Aurora Serverless v2 (最小 0.5 ACU)** | $50〜80/月 | コールドスタートでレイテンシ増 |
| **ワーカーノードを Spot Instance** | EC2 費用 60〜90% 減 | Pod が突然終了する。学習にはむしろ良い題材 |
| **Datadog → CloudWatch Container Insights** | $30〜60/月 | UI / 機能性は劣る |
| **平日夜・週末のみ起動 (cluster autoscaler で 0 ノード)** | 50〜70% 減 | 触る時しか使えない |
| **検証時のみ `terraform apply` / 不要時 `destroy`** | 最大 90% 減 | state 管理に慣れる学習にはなる |

### 5.2 強く推奨

**「触らないときは destroy する」運用** が最も効く。Terraform で構築する最大の利点はここで、`terraform destroy` で全削除 → 触る時に `apply` で再構築、というサイクルを回せば **月額を 1/5〜1/10 に抑えられる**。

ただし以下は破棄/再作成すると面倒なので、Terraform module を「**破棄しない層**」と「**破棄する層**」に分けて構成する:

- ECR のイメージ (再 push が必要)
- Route53 のホストゾーン (DNS 委任が崩れる)
- Aurora のデータ (snapshot は別途取る)
- ACM 証明書 (再発行で DNS 検証が走る)

## 6. 学習目的を外した場合の代替案 (4 パターン比較)

「K8s 学習目的を外して、純粋にこのアプリを安く本番運用したい」場合の選択肢を整理する。**単一サービスへの置き換えではなく、ワークロード特性ごとに基盤を分離するハイブリッド構成が本命**。

### 6.1 比較サマリ (想定中央値: MAU 10 万、月間 20M req)

| 候補 | 月額 (USD) | スケール耐性 | 改修コスト | 学習価値 |
|------|----------|-----------|---------|--------|
| **🥇 ハイブリッド (ECS Fargate + Lambda)** | **$280〜350** | ◎ MAU 35万まで対応可 | 軽微 | ○ |
| **🥈 App Runner + Lambda** | $200〜350 | △ 数万 MAU まで | 軽微 | 中 |
| **🥉 Lambda 全載せ + DynamoDB** | $80〜200 | ○ ただし RDS Proxy 等の制約 | **大** (データモデル書き換え) | ○ |
| **ECS Fargate 単独 (api/job 常時)** | $250〜400 | ○ | なし | 中 |

> 代替案は **NAT Gateway を 1 個に集約する前提** で試算。3 AZ 構成にして冗長化する場合は +$66/月 を上乗せ。EKS と異なり、ECS / App Runner / Lambda は単一 NAT GW でも実用上問題が出にくい (タスクの可搬性が高い)。

### 6.2 本命: ハイブリッド構成 (ECS Fargate + Lambda)

```
[フロント]
CloudFront ─ S3 (Next.js 静的出力)
       │
       └─ ALB ─ ECS Fargate (api タスク × 2-6, Auto Scaling)
                       │
                       ├─ Aurora Serverless v2 (0.5-4 ACU)
                       └─ SQS

[非同期]
EventBridge Scheduler ─ Lambda (batch)        ← CronJob 置き換え
SQS ─ Lambda (job worker)                     ← Redis ワーカー置き換え
```

#### 月額内訳 (中央値想定)

| 項目 | 月額 (USD) |
|-----|-----------|
| ECS Fargate (api 0.5 vCPU × 3 task avg) | $80 |
| ALB | $25 |
| NAT Gateway × 1 | $33 |
| Aurora Serverless v2 (avg 1.5 ACU) | $135 |
| Lambda (batch, 月 1000 回起動) | $1 |
| Lambda (job, 月 100 万メッセージ処理) | $5〜10 |
| SQS | $0.4 (月 100 万) |
| CloudFront / S3 | $5 |
| CloudWatch | $10 |
| **合計** | **約 $300/月** |

#### この分離が効く理由

| ワークロード | 適する基盤 | 理由 |
|------------|----------|-----|
| **api (リアルタイム HTTP)** | コンテナ (ECS Fargate) | UX 上のレイテンシ要件、接続プール、常時稼働の経済性 |
| **batch (定期実行)** | Lambda + EventBridge | 起動間隔が空く = アイドル時間帯への課金がもったい |
| **job (キュー駆動)** | Lambda + SQS | メッセージ駆動 = リクエスト駆動と本質的に同じ。ピーク自動追従 |

→ EKS の **約 60% (40〜50% off)** に圧縮しつつ、production-grade な構成を維持できる。

### 6.3 軽量代替: App Runner + Lambda

立ち上げ期 (MAU 数万まで) なら、ECS Fargate を **App Runner** に置き換えるとさらに簡素化できる:

- 既存 Dockerfile がそのまま動く
- ALB 不要 (HTTPS エンドポイント自動付与)
- `min instance = 0` でアイドル時の課金を 0 にできる
- ただし **デフォルト 25 instance 上限**、WebSocket 制約等あり

→ **MAU 数万まではコスト・運用性で優れる。スケール期は ECS Fargate に切り替え**、という二段構えが妥当。

### 6.4 Lambda 全載せ (非推奨パス)

API 含めて全て Lambda 化する案 ([Lambda Web Adapter](https://github.com/awslabs/aws-lambda-web-adapter) 利用):

| 利点 | 欠点 |
|-----|-----|
| 絶対コスト最安 ($80〜200) | コールドスタートが UX に影響 |
| 完全リクエスト駆動 | RDS Proxy 必須 ($15) + 接続管理複雑化 |
| 同時実行制限管理が必要 | サスティンドな高負荷で割高に転じる |

→ **B2C で UX 重視** + **想定トラフィック** を踏まえると不利。**短時間バースト主体のサービス向け**。

### 6.5 EKS と各代替案の比較表

| 観点 | EKS (現状) | ハイブリッド | App Runner | Lambda 全載せ |
|-----|-----------|-----------|----------|------------|
| **月額固定費 (中央値)** | $464〜564 | $280〜350 | $200〜350 | $80〜200 |
| **スケール上限 (MAU)** | 無制限 | 35万+ | 数万 | 数万 |
| **コールドスタート** | なし | なし | min=0 時のみ | あり |
| **学習価値 (K8s)** | ◎ | × | × | × |
| **学習価値 (AWS)** | ○ | ○ | ○ | ○ |
| **マルチクラウド適性** | ◎ | × | × | × |
| **エコシステム** | Helm/Argo/KEDA/Istio | AWS 標準 | AWS 標準 | AWS 標準 |
| **転職市場での評価** | 高 | 中 | 中 | 中 |

### 6.6 推奨される判断軸

| 状況 | 推奨 |
|-----|-----|
| K8s スキルが転職・キャリアの目的にある | **EKS のまま継続** + 節約策併用 |
| プロダクト本格運用を優先・コスト圧縮重視 | **ハイブリッド (ECS Fargate + Lambda)** |
| 立ち上げ期のみ最小コストで動かしたい | **App Runner + Lambda** (MAU 数万でハイブリッドに移行) |
| 両方触りたい | EKS を 3〜6 ヶ月運用 → 学んだ後ハイブリッドに移行 |

## 7. まとめ

- **回収可能性**: 想定ユーザーモデル (社会人 + 学生) を字面通り受ければ、**MAU 10 万到達時点でフリーミアムにより回収可能**。ただし立ち上げ期は赤字運用
- **オーバーエンジニアリング判定**: ライフサイクル依存。**立ち上げ期は過剰、スケール期は適正**。学習目的では全フェーズで正当化される
- **コスト**: 月額 $464〜564 が固定費 (3 AZ 構成)。**節約策 (destroy 運用 / Spot / Aurora Serverless / 脱 Datadog / NAT GW 集約) で 1/5〜1/10 まで圧縮可能**
- **代替案**: 学習目的を外した場合、**ECS Fargate (api) + Lambda (batch/job) のハイブリッド構成 (約 $300/月)** が本命。App Runner + Lambda は立ち上げ期向けの軽量代替

**本プロジェクトを進めるにあたっては、「学習プロジェクトであり、本番 SaaS としての収益化を目的としない」と明示的に位置づけた上で、立ち上げ期は節約策を併用しつつ、K8s の学習価値を最大化する** 進め方が健全となる。プロダクトをスケールさせるフェーズに入った段階で、改めてアーキテクチャの再評価 (ハイブリッド構成への移行可否) を行う想定。
